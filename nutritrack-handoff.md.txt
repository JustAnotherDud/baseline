# NutriTrack — Documento de Contexto

## O que é o NutriTrack

PWA de tracking de nutrição construída de raiz para o José (Dud), corredor de endurance com turnos variáveis no Lidl. Substituição do MyFitnessPal — mais aberto, com API acessível, e integrado no ecossistema sync_hub/DCB.

A app corre em **GitHub Pages** (estática, sem servidor), usa **Supabase** como base de dados, e está integrada com o **sync_hub** via MCP para que o DCB leia e escreva dados de nutrição automaticamente.

## Schema Supabase

### Tabela `foods`
```sql
id               bigint PK
name             text NOT NULL
brand            text
serving_size_g   numeric          -- porção habitual, pré-preenche o campo de gramas
calories_per_100g numeric NOT NULL
protein_per_100g  numeric NOT NULL
carbs_per_100g    numeric NOT NULL
fat_per_100g      numeric NOT NULL
saturated_fat_per_100g numeric default 0
sugar_per_100g    numeric default 0
fiber_per_100g    numeric default 0
created_at        timestamptz
```

### Tabela `diary`
```sql
id            bigint PK
date          date NOT NULL default current_date
logged_at     timestamptz default now()
meal          text NOT NULL   -- ver MEALS abaixo
food_id       bigint → foods(id) (nullable — quick adds não têm food_id)
food_name     text NOT NULL   -- snapshot do nome (intencional)
grams         numeric         -- null para quick adds
calories      numeric NOT NULL
protein       numeric NOT NULL
carbs         numeric NOT NULL
fat           numeric NOT NULL
saturated_fat numeric default 0
sugar         numeric default 0
fiber         numeric default 0
```

### Tabela `daily_targets` ← tabela activa (lida pela PWA)
```sql
id            bigint PK
date          date NOT NULL UNIQUE
calories      numeric NOT NULL
fat           numeric NOT NULL
saturated_fat numeric NOT NULL
carbs         numeric NOT NULL
sugar         numeric NOT NULL
fiber         numeric NOT NULL
protein       numeric NOT NULL
blocks_active jsonb             -- blocos activos enviados pelo DCB (ex: {base:1800,run:400})
updated_at    timestamptz
```

### Tabela `targets` ← DEPRECATED (não lida pela PWA)
```sql
id            bigint PK
day_type      text NOT NULL UNIQUE  -- rest_pure | work_only | training_only | …
calories      numeric NOT NULL
fat           numeric NOT NULL
saturated_fat numeric NOT NULL
carbs         numeric NOT NULL
sugar         numeric NOT NULL
fiber         numeric NOT NULL
protein       numeric NOT NULL
updated_at    timestamptz
```
> **Nota:** A PWA lê exclusivamente de `daily_targets` (chave por `date`).
> A tabela `targets` (chave por `day_type`) está deprecated — o antigo
> `sync_hub_push_targets` escrevia aqui; o actual `sync_hub_push_daily_target`
> escreve em `daily_targets`. Não escrever novos dados em `targets`.

## Constantes importantes

### MEALS (refeições)
```js
breakfast:  'Pequeno-almoço'
morning:    'Lanche manhã'
lunch:      'Almoço'
afternoon1: 'Lanche tarde 1'
afternoon2: 'Lanche tarde 2'
dinner:     'Jantar'
supper:     'Ceia'
```

### Auto-selecção de refeição por hora
```
6h-10h  → breakfast
10h-12h → morning
12h-15h → lunch
15h-18h → afternoon1
18h-20h → afternoon2
20h-23h → dinner
23h-6h  → supper
```

### Day types (targets)
```
rest_pure | work_only | training_only | training_plus_work | race_or_intense | custom
```

### Indicadores de cor por nutriente
| Nutriente | Verde | Amarelo | Vermelho |
|---|---|---|---|
| Calorias, Hidratos | 90–110% | 80–90% / 110–120% | <80% / >120% |
| Proteína | 90–130% | 80–90% / 130–150% | <80% / >150% |
| Gordura total | 50–120% | 30–50% / 120–140% | <30% / >140% |
| Gord. Saturada, Açúcar | ≤85% | 85–100% | >100% |
| Fibra | ≥90% | 70–90% | <70% |

## Integração sync_hub / DCB

### Ferramentas MCP disponíveis
```python
sync_hub_nutrition_fetch(target_date: str)
# Lê o diário do Supabase para uma data. Devolve totais e breakdown por refeição.
# Output guardado em D:\sync_hub\output\nutrition_today.json

sync_hub_push_targets(day_type, calories, fat, saturated_fat, carbs, sugar, fiber, protein)
# Upsert na tabela targets do Supabase.
# A PWA lê os targets desta tabela — DCB actualiza, app reflecte automaticamente.
```

### Fluxo DCB → NutriTrack
1. DCB determina `day_type` do dia (trabalho calendar + plano de treino)
2. Chama `sync_hub_push_targets` com os valores correctos
3. PWA lê da tabela `targets` no próximo reload — sem intervenção do utilizador

## Como fazer deploy

O projecto usa **Claude Code** para editar e fazer push:

```bash
cd C:\Users\josef\nutrition-tracker
claude
```

Dentro do Claude Code:
- Trabalhar sempre no branch `main` directamente
- Fazer commit e push no final de cada sessão de trabalho
- A GitHub Pages faz deploy automático após push (1-2 min)
- Após deploy: Ctrl+Shift+R no browser para limpar cache

## Decisões de arquitectura

**Script tags em vez de ES modules** — sem build step, sem bundler, compatibilidade máxima para GitHub Pages estático.

**Snapshot intencional no diary** — os valores nutricionais são copiados para cada entrada do diário em vez de calculados a partir do alimento. Histórico não muda quando um alimento é corrigido. Trade-off aceite.

**Targets em Supabase em vez de localStorage** — permite que o DCB faça push de targets sem o utilizador interagir com a app. localStorage como cache local para arranque rápido.

**Quick add separado da DB** — entradas de restaurante/estimativas não criam alimentos na base de dados. `food_id` é null, `grams` é null.

**Sem RLS** — projecto pessoal, uso único. Publishable key não é secret, mas sem RLS qualquer pessoa com a key pode ler/escrever. Aceitável para uso pessoal sem dados sensíveis.

## Notas de desenvolvimento

- Verificar sempre sintaxe JS antes de fazer push — extrair script com Python + `node --check`
